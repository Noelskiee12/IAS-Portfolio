AIS NOTES

CWE-89: SQL INJECTION

— Attack where malicious SQL is inserted into input fields to manipulate database queries, allowing data theft, modification, or deletion and possible full system compromise.

IMPACT AND CONSEQUENCES

— Data breaches, unauthorized data modification, loss of integrity, leaked credentials, financial/legal penalties, reputational damage, and potential full takeover of the application or backend systems.

MITIGATION STRATEGIES

— Use parameterized queries, input validation, least-privilege DB accounts, stored procedures, escaping where needed, web application firewalls, regular code reviews, and security testing (SAST/DAST/penetration tests).

Prepared Statements

— Queries where SQL and data are separated; placeholders are used for inputs so the database treats them as data, not code.

Example: SELECT * FROM users WHERE id = ?

Input Validation

— Check and restrict user input by type, length, format, and allowed characters. Prefer allowlists over deny lists. Reject or sanitize invalid input before it reaches the database.

Stored Procedures

— Predefined database routines that encapsulate logic and reduce dynamic SQL use. Effective if implemented safely, but still require parameterization and proper privileges.

Escaping User-Supplied Input

— Replace or prefix special SQL characters so input cannot close literals or inject SQL. Use only as a last resort — parameterized queries are safer and preferred.
